# TextDrop 产品需求描述文档 v0.1

## 1. 产品名称

**TextDrop**

## 2. 产品定位

TextDrop 是一个轻量级跨设备输入工具。

用户可以在手机浏览器中输入文字，或使用手机输入法的语音输入能力将语音转成文字，然后点击发送按钮，将文本直接发送到 Windows 电脑当前光标所在位置。

第一阶段只支持局域网场景。

---

## 3. 核心使用场景

用户在 Windows 电脑前工作，但电脑麦克风距离较远，不适合直接语音输入。

用户希望使用手机作为近距离输入端，通过手机上的任意输入法完成语音转文字，再将文字快速发送到电脑当前正在编辑的位置。

典型场景：

* 在 ChatGPT、Claude、DeepSeek 等网页输入框中快速输入长文本
* 在飞书、微信、Word、Notion、思源笔记等应用中输入文字
* 在搜索框、表格备注、代码注释、文档草稿中快速输入
* 使用手机输入法的语音识别能力替代电脑麦克风输入

---

## 4. 第一阶段目标

TextDrop v0.1 只解决一个核心问题：

> 手机浏览器输入文本，点击发送后，文本自动粘贴到 Windows 当前光标位置。

第一阶段不做云端同步、不做账号系统、不做历史记录、不做自动整理、不做语音识别能力。

Windows 端第一阶段采用桌面端 GUI 形态。程序需要能够打包为可发布的便携版程序，用户运行时不应依赖额外安装 Python、Node.js 或其他运行环境。

第一阶段发布形态采用 zip 压缩包。压缩包内包含 `TextDrop.exe` 和运行所需依赖目录，用户首次使用前解压整个文件夹，然后运行文件夹内的 `TextDrop.exe`。

第一阶段技术栈采用 Python + PySide6 + FastAPI/uvicorn + PyInstaller。

---

## 5. 用户角色

### 5.1 Windows 端用户

负责启动 TextDrop，查看连接地址和二维码，保持电脑端程序运行。

### 5.2 手机端用户

通过手机浏览器访问 TextDrop 页面，在文本框中输入内容，并点击按钮发送到电脑。

---

## 6. 核心流程

### 6.1 首次使用流程

1. 用户在 Windows 电脑上启动 TextDrop。
2. TextDrop 显示局域网访问地址。
3. TextDrop 显示二维码。
4. 用户用手机扫码打开手机端网页。
5. 手机端页面显示输入框和发送按钮。
6. 用户在电脑上点击目标输入框，使光标停留在目标位置。
7. 用户在手机端输入文本。
8. 用户点击“发送到电脑”。
9. 电脑端将文本粘贴到当前光标位置。
10. 手机端发送成功后自动清空文本框。

### 6.2 后续使用流程

1. 用户启动 Windows 端 TextDrop。
2. 手机可以通过之前保存的页面、浏览器历史、书签或桌面快捷方式打开。
3. 如果电脑局域网地址没有变化，用户无需重新扫码。
4. 如果地址变化，用户重新扫码或手动输入新地址。

---

## 7. 功能需求

## 7.1 Windows 端功能

### 7.1.1 启动服务

用户打开 TextDrop 后，程序进入运行状态，并允许手机端连接。

启动要求：

* 双击程序后自动打开 Windows 主界面
* 程序启动后自动启动本地服务
* 如果默认端口不可用，自动寻找可用端口并使用实际端口生成访问地址
* 程序主窗口按正常桌面程序方式开发，不要求固定窗口大小
* 第一阶段不做开机自启动
* 第一阶段不做系统托盘常驻

界面需要展示：

* 产品名称：TextDrop
* 当前运行状态
* 手机访问地址
* 二维码
* 访问 Token 状态
* 语言选择
* 简单操作按钮

---

### 7.1.2 显示访问地址

Windows 端需要显示手机可访问的局域网地址。

示例：

```text
http://192.168.1.23:8765/?token=xxxx
```

要求：

* 地址应清晰可复制
* 地址应包含访问 Token
* 地址用于手机浏览器手动访问
* 服务监听 `0.0.0.0`，允许同一局域网内设备访问
* 界面展示自动选择的局域网 IPv4 地址
* 电脑存在多个网络地址时，优先选择手机更可能访问到的物理网络地址，例如 Wi-Fi 或有线网卡地址
* 自动选择地址时应尽量排除 VPN、系统代理、虚拟机、容器、回环网卡等不适合手机访问的地址
* Windows 主界面提供地址下拉框，允许用户在多个候选局域网地址之间手动切换
* 用户切换地址后，应立即更新访问地址和二维码
* 如果端口自动切换，界面应展示实际可访问端口
* Token 或端口变化时，应立即重新生成访问地址和二维码
* 语言变化不改变访问地址，不需要重新生成二维码

---

### 7.1.3 显示二维码

Windows 端需要根据访问地址生成二维码。

要求：

* 手机扫码后直接打开输入页面
* 二维码内容应包含完整访问地址和 Token
* 用户无需手动输入 Token

---

### 7.1.4 复制地址

Windows 端提供“复制地址”按钮。

点击后，将完整手机访问地址复制到剪贴板，便于用户通过微信、飞书等方式发送到手机。

---

### 7.1.5 接收手机端文本

Windows 端需要接收手机端发送的文本内容。

接收成功后，应立即将文本输入到当前光标所在位置。

要求：

* 只处理文本
* 不处理图片、文件、富文本
* 不额外追加回车、空格、换行
* 不删除、不裁剪、不规范化用户输入内容
* 用户输入什么就发送什么，包括空内容、空格、换行和多行文本
* 不做文本改写、润色、翻译或摘要
* 第一阶段不在粘贴后自动按 Enter
* 第一阶段不自动提交目标应用中的发送按钮
* 后续版本计划提供可选的粘贴后自动按键功能
* 单次发送内容大小上限为 100KB
* 超过上限时拒绝发送，并在手机端保留原文本

---

### 7.1.6 粘贴到当前光标位置

Windows 端收到文本后，需要将文本粘贴到当前系统光标所在位置。

要求：

* 用户应先在电脑上选中目标输入框
* TextDrop 不负责选择目标输入框
* TextDrop 不弹出额外确认窗口
* TextDrop 不应主动抢占焦点
* 粘贴内容应与手机端文本一致
* 第一阶段采用“写入系统剪贴板 + 模拟 Ctrl+V”的方式触发粘贴
* TextDrop 不检测当前光标是否位于可输入位置
* TextDrop 不检测目标应用是否实际写入成功

---

### 7.1.7 剪贴板处理

第一阶段不需要保留原剪贴板内容。

允许 TextDrop 在粘贴过程中覆盖系统剪贴板。

要求：

* 不需要粘贴后恢复原剪贴板
* 不需要提示剪贴板被覆盖
* 不需要剪贴板历史
* 复制访问地址和发送文本到电脑都会使用系统剪贴板

---

### 7.1.8 Token 管理

TextDrop 需要使用访问 Token 做基础安全控制。

要求：

* 首次启动生成一个随机 Token
* Token 保存在电脑本地配置中
* Windows 配置默认保存到 `%APPDATA%\TextDrop\config.json`
* 后续默认复用该 Token，减少重复扫码
* Windows 端提供“刷新 Token”按钮，由用户手动触发
* 点击“刷新 Token”后需要二次确认
* 确认文案需要提示：刷新后旧链接将失效，请再次扫码连接
* Token 变化后，旧手机页面失效
* 新二维码和访问地址应使用新 Token
* Windows 主界面不单独明文展示 Token
* 完整访问地址和二维码中包含 Token

---

### 7.1.9 运行状态显示

Windows 端只展示基础运行状态：

* 正在运行
* 已关闭

状态提示应简洁，不需要日志界面。

Windows 端不显示用户发送的正文预览，不弹出发送成功通知。用户通过目标输入位置是否出现文本来判断实际输入效果。

---

### 7.1.10 退出程序

Windows 端提供退出入口。

退出后：

* 手机端无法继续发送文本
* 再次启动后恢复服务
* 如果 Token 未被重置，应继续使用原 Token
* 关闭窗口即退出程序
* 退出后不在系统托盘继续运行

---

## 7.2 手机端功能

### 7.2.1 手机浏览器访问

手机端不开发原生 App，第一阶段使用浏览器网页。

要求：

* 安卓浏览器可访问
* iOS 浏览器理论上也可访问，但第一阶段不重点适配
* 页面适合手机竖屏使用
* 页面打开后即可输入

---

### 7.2.2 文本输入框

手机端页面需要提供一个大文本框。

要求：

* 支持多行输入
* 适合长段文字输入
* 支持手机输入法调用语音输入
* 用户可以在发送前检查、修改文本
* 输入框应占据页面主要区域

---

### 7.2.3 发送按钮

手机端提供明确的“发送到电脑”按钮。

要求：

* 只有点击按钮才发送
* 不支持回车发送
* 不支持自动发送
* 不支持手势发送
* 按钮应明显、易点击
* 避免误触

---

### 7.2.4 发送成功后自动清空

当手机端文本成功发送到电脑后，自动清空当前文本框。

发送成功定义：

* Windows 端收到文本
* Windows 端成功写入系统剪贴板
* Windows 端成功触发粘贴快捷键
* 手机端收到成功回执

第一阶段发送成功后不额外触发 Enter，不自动提交目标应用中的输入框。

要求：

* 成功后立即清空
* 不需要保留历史
* 不需要二次确认

---

### 7.2.5 发送失败保留文本

如果发送失败，手机端必须保留文本框内容。

失败情况包括但不限于：

* 电脑端未运行
* 网络断开
* Token 无效
* 请求超时
* 电脑端处理失败

要求：

* 失败时不清空文本
* 页面给出简短失败提示
* 用户可以再次点击发送

---

### 7.2.6 清空按钮

手机端可以提供“清空”按钮。

要求：

* 用于手动清空当前文本框
* 位置不应过于突出，避免误触
* 点击后可以直接清空，不需要复杂确认

---

### 7.2.7 连接状态提示

手机端需要展示简单连接状态。

状态包括：

* 连接成功
* 未连接

状态文案应简洁。

连接检测要求：

* 手机页面打开后先检查与电脑端的连接状态
* 手机页面每 5 秒自动检测一次连接状态
* 每次发送前先检查连通性
* 连接成功则继续发送
* 连接失败则提示未连接
* 未连接时不禁用发送按钮，用户可以再次点击发送并重新检测
* 连接检测和发送请求的超时时间为 3 秒
* 连接检测不保存、不上传、不处理输入框正文

---

## 8. 安全需求

## 8.1 访问 Token

第一阶段只使用访问 Token 作为基础安全机制。

要求：

* 手机端请求必须携带 Token
* Token 不正确时拒绝发送
* Token 不正确时手机端显示失败提示
* Token 应足够随机，避免被猜中

---

## 8.2 局域网边界

第一阶段默认 TextDrop 只用于可信局域网环境。

不处理复杂安全场景：

* 公网访问
* 多用户账号
* 设备权限管理
* 端到端加密
* 公司级权限控制
* 审计日志

---

## 8.3 文本存储

第一阶段不保存用户发送的文本。

要求：

* 手机端不保存历史记录
* Windows 端不保存接收历史
* 程序不记录发送事件日志
* 程序不记录正文内容
* 本地仅保存运行所需配置，例如 Token 和语言设置

---

## 9. 非功能需求

## 9.1 简洁性

TextDrop v0.1 应保持极简。

不做超出核心流程的功能。

核心原则：

> 能发送文本到当前光标即可。

---

## 9.2 低打扰

Windows 端不应干扰用户正在使用的应用。

要求：

* 不主动弹窗抢焦点
* 不切换当前窗口
* 不要求用户在电脑端确认
* 收到文本后直接粘贴

---

## 9.3 易连接

连接过程应尽量少操作。

要求：

* 二维码优先
* 支持手动输入地址
* Token 默认长期保留
* 电脑地址变化时允许重新扫码
* 不要求每次启动都重新配对

---

## 9.4 响应速度

发送后应尽快在电脑端完成粘贴。

目标体验：

> 点击发送后，用户应感觉文本几乎立即出现在电脑当前输入框。

---

## 9.5 兼容性

第一阶段重点支持：

* Windows 电脑
* 手机浏览器
* 同一局域网环境
* 可独立运行的 Windows 文件夹程序
* zip 压缩包发布，解压后以文件夹形式运行
* 电脑连接公司 Wi-Fi、家庭 Wi-Fi 或手机热点
* 电脑同时开启 VPN、系统代理或虚拟网卡时，仍优先支持局域网手机访问

不保证：

* macOS
* Linux
* 云端连接
* 跨网络连接
* 多电脑同时连接
* 多手机同时连接
* 后台常驻 App 体验

---

## 9.6 多语言

TextDrop v0.1 默认使用中文界面，并保留多语言开发能力。

要求：

* Windows 端支持中文和英文界面
* 手机端支持中文和英文界面
* 默认语言为中文
* 语言在 Windows 端选择
* Windows 主界面使用下拉框选择语言，选项为“中文”和“English”
* 切换语言后 Windows 主界面立即生效
* 手机端页面语言跟随 Windows 端选择
* 手机端刷新页面后使用最新语言
* 第一阶段不需要自动识别系统语言

---

## 9.7 技术与配置约束

第一阶段采用以下技术方案：

* Windows GUI：PySide6
* 本地服务：FastAPI/uvicorn
* 打包发布：PyInstaller
* 运行形态：便携版 Windows 文件夹
* 发布形态：zip 压缩包
* 第一阶段不制作安装包

配置保存要求：

* 配置文件保存到 Windows 用户配置目录
* 默认路径为 `%APPDATA%\TextDrop\config.json`
* 配置至少包含 Token 和语言设置

---

## 10. 不做的功能

TextDrop v0.1 明确不做以下功能：

* 不做云端连接
* 不做账号系统
* 不做登录注册
* 不做手机原生 App
* 不做安卓输入法
* 不做语音识别
* 不做 AI 润色
* 不做文本翻译
* 不做历史记录
* 不做剪贴板恢复
* 不做多设备管理
* 不做快捷短语
* 不做文件传输
* 不做图片传输
* 不做富文本格式
* 不做端到端加密
* 不做浏览器插件
* 不做开机自启动
* 不做系统托盘常驻
* 不做目标输入框识别
* 不做目标应用写入结果检测
* 不做安装包
* 不做手机端本地草稿保存

## 11. 页面与界面要求

## 11.1 Windows 端主界面

建议包含：

```text
TextDrop

状态：正在运行

语言：[中文 v]

粘贴后按键：[不自动按键 v]

手机访问地址：
http://192.168.x.x:8765/?token=xxxx

[二维码]

[复制地址] [刷新 Token] [退出]
```

状态区域只显示运行状态：

```text
正在运行
已关闭
```

粘贴后按键为可选设置，默认"不自动按键"。用户可选择 Enter、Ctrl+Enter、Shift+Enter 或 Alt+Enter。选择后每次粘贴完成会自动模拟按下对应按键。配置会持久化保存。

---

## 11.2 手机端页面

建议包含：

```text
TextDrop

连接状态：连接成功        [清空]

[大文本框]

[发送到电脑]
```

实际布局要求：

```text
[连接状态]      TextDrop      [清空]

[大文本框]

[发送到电脑]
```

页面风格：

* 简洁
* 大输入框
* 大按钮
* 适合单手操作
* 不需要复杂导航
* 不需要设置页
* 不保存本地草稿，浏览器刷新或页面关闭后未发送文本可以丢失
* 顶部左侧显示连接状态，使用文字和颜色标注
* 顶部中间显示产品名 TextDrop，产品名应绝对居中
* 顶部右侧放置清空按钮，视觉弱化，避免误触
* 中间区域为大文本框，占据页面主要空间
* 发送按钮放在页面底部，占据整行，明显且易点击
* 键盘弹起时，底部发送按钮应跟随可视区域上移

---

## 12. 关键交互细节

## 12.1 手机发送空文本

第一阶段允许发送空文本。

要求：

* 程序不判断文本是否为空
* 程序不去除空格或换行
* 空文本发送成功后仍按成功流程清空输入框
* 空文本实际粘贴效果为空，用户通常不会感知到输入变化

---

## 12.2 发送过程中重复点击

发送过程中应避免重复发送。

要求：

* 发送按钮进入临时不可点击状态
* 发送完成后恢复
* 如果失败，文本保留

---

## 12.3 Token 无效

如果手机端 Token 无效，发送失败。

提示：

```text
连接已失效，请重新扫码
```

---

## 12.4 电脑端未运行

如果电脑端未运行，手机端发送失败。

提示：

```text
发送失败，请确认电脑端正在运行
```

---

## 12.5 电脑光标不在输入框

如果用户电脑当前没有可输入位置，TextDrop 仍尝试粘贴。

第一阶段不做复杂判断。

用户需要自行确保：

> 发送前先把电脑光标放到目标输入框。

如果目标位置不支持文本输入，TextDrop 仍可能返回发送成功，因为第一阶段只判断是否成功触发粘贴动作，不判断目标应用是否实际接收文本。

---

## 13. 成功标准

TextDrop v0.1 视为成功，需要满足：

1. Windows 端可以启动并显示访问地址。
2. Windows 端可以显示二维码。
3. 手机扫码后可以打开输入页面。
4. 手机端可以输入多行文本。
5. 点击发送后，Windows 端收到文本并成功触发粘贴动作。
6. 发送成功后手机文本框自动清空。
7. 发送失败时手机文本框保留内容。
8. Token 错误时无法发送。
9. 刷新 Token 后，旧链接失效。
10. 端口冲突时可以自动切换到可用端口。
11. 第一阶段无需任何账号、云服务或手机 App。
